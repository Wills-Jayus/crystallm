二、典型报错定位与解决（命令行/流程问题）
------------------------------------------------------------
A. prompt_optimization_loop.py 命令报错（反斜杠续行）
- 现象：prompt_optimization_loop.py 提示 unrecognized arguments，bash 报 --rounds: command not found。
- 根因：行尾的续行符 "\" 后面多了一个空格，导致 bash 不再续行，后续参数被当成新命令。
- 解决：确保每一行末尾的 "\" 后不带任何空格，或将命令写成单行。

B. sample.py 报错 top_p 不支持
- 现象：KeyError: "'top_p' is not a supported option"。
- 根因：bin/sample.py 的配置 dataclass SampleDefaults 不包含 top_p 字段，parse_config 不接受该 override。
- 解决：去掉 top_p 参数；若确需 top_p，需要修改 SampleDefaults 与 generate 调用链以支持。

C. evaluate_cifs.py 报错 FileNotFoundError: 'cifs'
- 现象：python bin/evaluate_cifs.py cifs --out tmp_eval.csv 报错。
- 根因：evaluate_cifs.py 期望输入为 .tar.gz 文件路径，而非目录名；且上一步 sample.py 失败也会导致输出文件不存在。
- 解决：先生成 sample_*.cif，再打包为 tar.gz，然后 evaluate。

D. lspci 命令缺失
- 现象：bash: lspci: command not found。
- 根因：容器未安装 pciutils。
- 解决：安装 pciutils（apt/yum），或改用 ip a / /proc/net/dev 获取网卡信息。


三、LLaMA（m3rg-iitd/llamat-3-chat）格式/模板排查与“乱码”定位
------------------------------------------------------------
A. 模型是否有输入输出模板要求？
- 通过本地 transformers 缓存中的 tokenizer_config.json（chat_template）确认：
  - 使用 Llama3 风格的 chat_template（<|start_header_id|>role<|end_header_id|>...<|eot_id|>）。
  - 仓库示例 run_llamat3_chat.py 使用 tokenizer.apply_chat_template 构造 prompt。
  - 结论：模型输入应采用 messages（system/user）→ apply_chat_template 展平；模型输出本身不强制 JSON，JSON 属于我们自定义协议。

B. 为什么闭环中 LLaMA 输出仍“乱码”？
- 现象（llama_output.json）：
  - output.analysis 固定为回退文案；next_prompt_lines 沿用上一轮；_raw 充满杂乱片段。
- 关键点：服务端 llama_local_api.py 会尝试解析 JSON；解析失败则回退并把 raw 塞入 _raw。
- 结论：
  - 乱码本质是模型生成的自由文本/回显/杂质 token；
  - 由于严格 JSON 协议（或模板协议）无法被模型稳定遵守，触发回退导致闭环提示不更新。

C. 设计并执行隔离实验验证上述判断

(1) 隔离实验一：JSON sanity
- 目录：/root/autodl-tmp/model/CrystaLLM/sandbox/llama_json_sanity
- 输入：payload_round1.json（包含 input: {last_prompt_lines, evaluator_summary}）
- 运行：温度 0.0/0.2/0.5 调用 bin/llama_single_test.py，输出保存到 runs/t*_response.log。
- 结果：三组均触发服务端 fallback JSON；_raw 仍混杂奇怪 token（如 ucwords/cement_composition/JADX PROXY 等），无法解析为干净 JSON。
- 结论：在最小输入和低温下仍无法稳定结构化输出，问题不依赖闭环复杂度。

(2) 隔离实验二：freeform（不要求 JSON 的普通问答）
- 目录：/root/autodl-tmp/model/CrystaLLM/sandbox/llama_freeform_test
- 输入：system="You are a helpful assistant..."，user="Explain band gap..."，temperature=0.2。
- 结果：模型在 _raw 中给出连贯的自然语言解释，但服务端仍尝试按 JSON 解析而回退，把自然语言塞进 _raw。
- 结论：模型本体能正常回答；主要冲突在于“服务端强制 JSON 解析 / 协议约束”与模型服从度不足。

四、物理化学过滤器（CIF 体检）模块梳理与文档
------------------------------------------------------------
A. 过滤器核心能力（当前版本）
- 基本检查：
  - 配方一致性（data / formula_sum / structural）
  - 位点多重度与元素计数一致性
  - 键长合理性评分（基于 CrystalNN 近邻，0~1）
  - 空间群一致性（声明 vs SpacegroupAnalyzer 检测）
- 清洗：
  - 针对 P4/<unk> 做兜底修正为 P4/mmm
  - 体积 sanity check
  - 对称操作替换（对合法空间群）
  - 去除 atom props block
- 输出：cif_quality.csv/json + summary.txt

B. 过滤结果如何进入闭环
- 每轮生成后先运行 CIF 质量检查；
- 清洗成功时覆盖写回 cif 文件，然后再送入 ALIGNN 打分；
- 打分表与质量表合并后，优先从“物理通过+打分成功”的结构中选 Top-K；
- 构造 evaluator_summary（包含通过率、物性统计、Top 样本失败原因）作为 LLaMA 输入。
