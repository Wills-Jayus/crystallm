0. 实验目标
-----------
在同一台机器上完成以下闭环实验能力：
1) CrystaLLM 生成若干 CIF 候选；
2) 过滤器（物理/晶体学检查）对 CIF 做清洗与验证；
3) ALIGNN 通过 ZMQ 服务对 CIF 预测 formation_energy、bandgap 等；
4) 由优化器 LLM 根据“上一轮 prompt + 本轮汇总摘要”产出下一轮 next_prompt_lines；
5) 循环多轮后，从全局候选中选择得分最好且通过验证的结构。

本次对话将优化器 LLM 从原先的 LLaMA 模型替换为 Qwen3，并用 vLLM 以 OpenAI API Server 方式拉起服务端。


1. 环境与资源（关键硬件/磁盘）
---------------------------
- 数据盘：/root/autodl-tmp（用于模型权重、缓存、日志与实验输出；避免落系统盘）
- GPU：NVIDIA A800 80GB（单卡）
- Python：3.10


2. Qwen3 服务端部署（vLLM OpenAI API）
------------------------------------
模型：
- Qwen/Qwen3-30B-A3B-Instruct-2507
- 本地路径：/root/autodl-tmp/models/Qwen/Qwen3-30B-A3B-Instruct-2507（约 57G）

关键点：
- vLLM 使用 OpenAI 兼容接口（/v1/models、/v1/chat/completions）。
- 启动后会常驻显存（模型权重 + KV cache 预留），空闲也会占用显存；不用时需 stop 释放显存。
- 为方便“需要时启动、不用时释放显存”，新增了一键启停脚本：
  - /root/autodl-tmp/qwen3_vllm/qwen3_vllmctl.sh
  - /root/autodl-tmp/qwen3_vllm/config.env（默认省显存参数：MAX_MODEL_LEN=8192、GPU_MEMORY_UTILIZATION=0.80）
  - 用法：start/stop/status/logs/test


3. ALIGNN 打分器上 GPU（关键修复点）
---------------------------------
在闭环实验中，出现过 ALIGNN 打分全失败（n_scored_ok=0），错误为：
  “Device API cuda is not enabled. Please install the cuda version of dgl.”

解决思路与结果：
- alignn-score 环境中 torch 为 cu121，但 dgl 之前是 CPU 版。
- 将 dgl 替换为 dgl-2.1.0+cu121 后，DGL 可在 cuda:0 上创建 graph。
- 重启 ALIGNN ZMQ 服务并验证 ZMQ 请求能返回数值预测：
  返回示例：{"ok": true, "formation_energy": -1.03..., "bandgap": 3.61...}

ALIGNN 服务：
- ZMQ 端口：tcp://127.0.0.1:5555
- 服务端脚本：/root/autodl-tmp/model/CrystaLLM/resources/alignn_zmq_server_multi.py
- 预测属性：formation_energy、bandgap


4. Qwen 接入闭环：输入输出协议（从“强约束”到“先跑通”）
----------------------------------------------------
原 LLaMA 优化器阶段曾因“严格 JSON/严格模板”导致输出不可控、解析失败、回退频繁。
本次接入 Qwen 的策略是：
- 先使用最宽松、最易跑通的约束（relaxed_v0）；
- 输入给模型的是“纯文本摘要”，避免让模型“回显/续写 JSON”；
- 输出不强制严格 JSON；只要模型能给出 next prompt lines，客户端做“宽松解析 + 清洗/合并”即可驱动下一轮。

Qwen 实际输入（每轮一次）：
- last_prompt_lines（上一轮 prompt.txt 的行列表）
- evaluator_summary（本轮汇总：样本数、打分成功数、通过率、bandgap/formation_energy 统计、top structures 的验证原因等）
说明：Qwen 不是对每个 CIF 各改一次 prompt，而是“每轮汇总一次 → 输出一次 next_prompt_lines”。

Qwen 实际输出（每轮一次）：
- 主要关注字段：next_prompt_lines（下一轮 prompt 行）
- 附带：简短的修改建议/分析文本（用于人工审计）
- 客户端会过滤掉不希望出现的细节行（如 _cell_length/_cell_angle），并把输出作为“补丁”合并回 last_prompt_lines，保证稳定性。


5. 实验目录与常见产物（每轮）
-------------------------
每个实验目录（out-dir）下通常包含：
- round_XX/prompt.txt：该轮真实输入给 CrystaLLM 的 prompt
- round_XX/cifs/sample_*.cif：该轮生成的 CIF（默认 samples_per_round 个）
- round_XX/cif_quality.json / .csv：过滤器输出（valid、reasons、bond_length_score、formula_ok、space_group_ok(三态)、multiplicity_ok 等）
- round_XX/cif_quality_summary.txt：过滤器汇总（通过率、space_group_ok 三态统计等）
- round_XX/scores.csv：ALIGNN 打分输出（formation_energy、bandgap）并合并过滤器关键字段
- round_XX/summary.json：本轮汇总（均值/极值、top_structures 等）
- round_XX/evaluator_summary.json：供 LLM 使用的压缩摘要
- round_XX/llama_output.json：本轮（非最后一轮）LLM 的输入输出记录（这里模型已换为 Qwen，但文件名沿用）
- final_report.json：全局汇总 + best_overall
- best_structures/：复制出来的候选（按 round/rank/score 命名）


6. 关键实验结果摘录（用于判断闭环是否“真跑通”）
--------------------------------------------
6.1 na2cl2_qwen_smoketest（早期冒烟）
- 现象：Qwen 输出可解析并更新了下一轮 prompt（证明“Qwen→next_prompt_lines→下一轮”链路通）
- 但当时 ALIGNN 因 DGL CUDA 未装对，打分全失败（n_scored_ok=0），导致难以评估“优化效果”。

6.2 na2cl2_qwen_v1_small（三轮闭环，打分已正常）
结果要点（来自 summary/final_report）：
- 每轮 ALIGNN 打分成功：total/ok=8/8（每轮）
- Round 01：
  - bandgap max≈4.2189
  - validation_ok 1/8
  - Qwen 修改：data_Na2Cl2 → data_Na2Cl3（空间群保持 P4/mmm）
- Round 02：
  - validation_ok 0/8（主要卡在 composition inconsistent / multiplicity inconsistent）
  - Qwen 修改：data_Na2Cl3 → data_Na2Cl2
- Round 03：
  - validation_ok 1/8
全局 best_overall（final_report.json）：
- round_01/cifs/sample_6.cif：bandgap≈4.2189 且 validation_ok=True（关键正样本）

观察：
- Qwen 输出稳定、能驱动闭环更新 prompt，但其“目标表述”曾出现与 max bandgap 不一致的措辞（提示后续需要在 system/user 中更明确写死 score_goal=max）。
- 过滤器通过率偏低，主要失败集中在化学计量一致性与多重度一致性。

6.3 na2cl2_qwen_v2_small（三态 space_group_ok 输出已用于产物）
- v2_small 的 cif_quality.json/scores.csv 中不再出现旧式的 validation_failed 直接中断式记录；
- space_group_ok 引入三态：True/False/None，并在 cif_quality_summary.txt 里汇总 true/false/none。

6.4 na2cl2_qwen_v4_small（失败原因分析的典型样例）
- Round 01：validation_ok 1/8；space_group_ok 3/1/4（true/false/none），常见失败为 composition inconsistent、space_group_check_error、bond lengths 不合理。
- Round 02/03：validation_ok 0/8；出现系统性 formula_check/strict_valid 错误（根因：Qwen 将 data_ 改成带小数形式 data_Na2.1Cl1.9，与 data_ 解析规则不兼容）。


7. 性能与运行体验（实验相关）
--------------------------
- vLLM Qwen3 服务启动后会占用大量显存（权重+KV cache），这是常驻服务的正常行为；不用时通过 stop 释放显存。
- 使用了省显存启动档位（max_model_len、gpu_memory_utilization）以便“按需启停”。
- ALIGNN 上 GPU 后，每轮 8 个 CIF 的打分可稳定返回数值（ok=8/8），闭环才具备可评估性。


8. 总结与可复现要点
-----------------
本次对话达成的“可复现闭环”关键点：
1) Qwen3 以 vLLM OpenAI API Server 方式稳定运行，并可用 curl 验证；
2) ALIGNN ZMQ 服务在 GPU 上稳定返回 formation_energy/bandgap；
3) Qwen 采用 relaxed 协议接入闭环：每轮读取 evaluator_summary + last_prompt_lines，输出 next_prompt_lines，驱动下一轮；
4) 过滤器输出可解析、不中断，space_group_ok 为三态并在 summary 中统计。

主要经验教训（对后续实验最重要）：
- LLM 输出“可跑通”优先：先宽松解析再逐步加约束，比一开始强制 JSON/模板更稳；
- 打分器必须先稳定（DGL CUDA 版本对齐），否则闭环只是在“盲改 prompt”；
- prompt 变更要避免与下游解析规则冲突（例如 data_ 带小数会触发系统性失败）；
- CrystaLLM 的元素约束目前偏弱，容易跑偏到非 Na/Cl，需要后续在 prompt 或后处理层加强约束策略。
