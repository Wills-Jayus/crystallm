你接手 SymCIF-v4 的 Mini-CFJoint 架构验证实验。

本轮只做架构验证，不追求一次性完整刷新 Table 3。核心问题是：

结构化多头联合模型 `P(W, A, X, L | formula, GT_SG)` 是否比当前分裂式 SymCIF-v4 pipeline 更适合 MP-20 的 `K<=5` 生成？

这里的术语含义：

- `W`: Wyckoff orbit / Wyckoff 位置骨架，即每个 crystallographic row 选哪个 Wyckoff 位置。
- `A`: atom assignment / 元素分配，即每个 Wyckoff row 放哪个元素。
- `X`: free parameters / 自由坐标参数，例如 x/y/z 中真正可变的维度。
- `L`: lattice / 晶格参数，即 a,b,c,alpha,beta,gamma。
- `GT_SG`: ground-truth space group / 真实空间群，本实验仍给模型使用。
- `K<=5`: 只看 top1 和 top5，不跑 top20。

当前背景：

- v3 CrystalFormer-like 失败的主要原因不是 W/A/X/L 分解思想错，而是 v3 仍是 GPT next-token 文本模型，不是真正的结构化多头、多 loss 模型。
- 之前 assignment exact-class retrieval 不可泛化，所以本实验不能把完整 assignment 当作一个类别。
- 当前 SymCIF-v4 已经有 OrbitEngine、stablekey、row free-param 修复、合法 OrbitToken、exact-cover feasibility、renderer/evaluator，可用于小规模联合模型验证。
- 当前 split pipeline 的强点是 WA constrained search，弱点主要在 `K<=5` WA 覆盖、geometry/free-param/lattice 质量，以及有 WA hit 但最终结构不 match 的样本。
- 本实验不是继续做 ranker/rerank replay，而是验证一个更接近 CrystalFormer/WyFormer 思想的小型 joint model 是否值得继续扩大。

硬性限制：

1. 只做 MP-20。
2. 只用 MP-20 train/val；test 不用于开发、调参或最终判断。
3. 不跑 MPTS-52。
4. 不跑 match@20。
5. 不启动 CrystaLLM baseline。
6. 不训练大模型。
7. 不同时开多个修正分支；每次只验证一个清楚假设。
8. 如果需要训练，尽量用较大 batch、num_workers、pinned memory、AMP 等方式提高 GPU/CPU 利用率；GPU 命令用非沙箱执行以确保 CUDA 可见。
9. 所有评估只报告：
   - match@1
   - match@5
   - RMSE@1
   - RMSE@5
   - WA_hit@1 / WA_hit@5
   - skeleton_hit@1 / skeleton_hit@5
   - readable@1/@5
   - formula_ok@1/@5
   - atom_count_ok@1/@5
   - SG_ok@1/@5
   - strict_valid@1/@5

必须复用现有 SymCIF-v4 资产：

- OrbitEngine
- stablekey canonicalization
- MP-20 structured data
- legal OrbitToken set
- formula exact-cover feasibility checker
- free_param extraction
- lattice normalization
- renderer/evaluator
- 当前 step-policy/action-mask 相关工具，能复用则复用

输出目录：

- 机器产物和中间表：`reports/symcif_v4_mp20_minicfjoint/`
- checkpoint/run 产物：`runs/symcif_v4_mp20_minicfjoint/`
- 最终人读实验报告：`Log_GPT/round_YYYYMMDD_NN_mp20_minicfjoint/`

最终报告必须放在 `Log_GPT`，不要只放在 `reports/`。

========================================
任务 0：preflight 与数据 roundtrip gate
========================================

在 MP-20 val 上审计 structured W/A/X/L target。

检查：

- structured record count
- failed records and reasons
- GT W/A/X/L -> OrbitEngine render roundtrip:
  - readable
  - formula_ok
  - atom_count_ok
  - SG_ok
  - strict_valid
- n_sites>=6 / n_sites>=12 / num_elements>=4 / extraction_hard breakdown

Gate:

如果 GT roundtrip 的 readable/formula_ok/atom_count_ok/SG_ok 任一低于 99%，停止，不训练模型，输出失败原因。

输出：

- `reports/symcif_v4_mp20_minicfjoint/00_preflight_roundtrip.md`
- `reports/symcif_v4_mp20_minicfjoint/00_preflight_roundtrip.json`

========================================
任务 1：构建 Mini-CFJoint dataset
========================================

构建结构化序列，不要文本化 CIF。

每个样本：

input:

- conventional cell formula embedding
- GT SG embedding

target rows:

- canonical row order
- legal `element@OrbitToken` action sequence
- row stop
- active free_params for each row

target lattice:

- normalized lattice parameters
- crystal-system-aware angle/length mask if available

关键要求：

- 主离散目标优先使用合法 `element@OrbitToken` action，而不是完全独立的 orbit_head + element_head。
- `element@OrbitToken` 的好处是一步动作同时约束元素、Wyckoff orbit、multiplicity 和 formula exact-cover。
- element assignment 是逐 row / 逐 action 生成，不是完整 assignment class。
- OrbitToken 必须来自当前 SG 的 legal set。
- free_params 只包含 active dimensions。
- stop 只能在公式完全闭合时作为合法动作。
- 保存 train/val dataset stats:
  - row count distribution
  - action vocab / orbit token vocab
  - active free_param dim distribution
  - element distribution
  - n_sites>=6 / n_sites>=12 / num_elements>=4 / extraction_hard stats

输出：

- `reports/symcif_v4_mp20_minicfjoint/01_dataset_audit.md`
- `reports/symcif_v4_mp20_minicfjoint/01_dataset_audit.json`

========================================
任务 2：实现小型 Mini-CFJoint 模型
========================================

实现一个小模型，不要大模型。

推荐模型形式：

- formula encoder
- SG embedding
- row autoregressive decoder，可以是 GRU 或小型 Transformer
- action head：预测下一个合法 `element@OrbitToken` 或 stop
- free_param head：预测当前 row 的 active free params
- lattice head：预测整结构 lattice parameters

不要第一版就把 orbit 和 element 完全拆成两个互不约束的 head。可以额外记录 orbit/element top1/top5 诊断，但训练主目标应优先对齐合法 action。

Loss:

`L_total =`

- `1.0 * action CE`
- `0.5 * wrapped free_param L1/L2`
- `0.5 * normalized lattice L1/L2`
- `0.2 * stop CE`，如果 stop 没有并入 action head
- 可选：`0.1~0.2 * row_count / remaining-count auxiliary`，只有实现简单且不会拖慢主线时才加

先不要实现复杂 mixture density / von-Mises。第一版以能稳定验证方向为主。

训练要求：

- train split 训练
- val split early stopping
- 小模型
- 记录 train/val loss curves
- 支持 AMP / larger batch / num_workers，提高 GPU/CPU 利用率
- 支持 small-overfit sanity

输出：

- `runs/symcif_v4_mp20_minicfjoint/`
- `reports/symcif_v4_mp20_minicfjoint/02_training_summary.md`

========================================
任务 2.5：small-overfit sanity，必须前置
========================================

在跑完整 val generation 之前，先做 128 或 256 train samples small-overfit。

目标：

- teacher-forcing action/orbit/element 几乎过拟合
- row_count / formula closure 在训练小集上接近完全正确
- GT W/A + predicted X/L render 可用
- full generation 在训练样本上明显 match

Gate:

- 如果 small-overfit 都失败，说明实现、target、mask 或 renderer 接口存在 bug，停止，不跑 full val。
- 如果 small-overfit 成功但 val 差，才说明是泛化、模型容量、loss 或数据复杂度问题。

输出：

- `reports/symcif_v4_mp20_minicfjoint/025_small_overfit_sanity.md`
- `reports/symcif_v4_mp20_minicfjoint/025_small_overfit_sanity.json`

========================================
任务 3：teacher-forcing 评估
========================================

在 MP-20 val 上评估 teacher-forcing 指标。

必须报告：

- action top1/top5
- orbit top1/top5，从 action 拆出诊断
- element top1/top5，从 action 拆出诊断
- row_count accuracy
- formula_closure rate under teacher-forcing
- free_param MAE / wrapped error
- lattice MAE
- n_sites>=6 / n_sites>=12 / num_elements>=4 / extraction_hard breakdown

目的：

先判断模型是否学到 W/A/X/L，而不是直接跑 end-to-end。

Gate:

- 如果 teacher-forcing action top5、row_count、formula closure 明显失败，不要进入任务 5 full generation。
- 这种情况下先根据 small-overfit 结果判断是实现 bug 还是泛化问题。

输出：

- `reports/symcif_v4_mp20_minicfjoint/03_teacher_forcing_eval.md`
- `reports/symcif_v4_mp20_minicfjoint/03_teacher_forcing_eval.json`

========================================
任务 4：GT-WA geometry controlled eval
========================================

只在 MP-20 val 上做：

GT W/A + Mini-CFJoint predicted X/L
-> OrbitEngine render
-> evaluator

第一版默认只报告 K=1，因为 GT W/A 固定时，普通确定性 free_param/lattice head 只产生一套结构。

只有在明确实现了连续变量的 5 个候选生成机制时，才允许报告 K=5。例如：

- dropout sampling
- Gaussian perturbation with fixed seed
- explicit uncertainty head
- ensemble-style repeated prediction

如果没有这些机制，不要伪造 K=5；报告 K=1，并在报告里说明 K=5 未定义。

必须输出：

- match@1
- RMSE@1
- readable@1
- formula_ok@1
- atom_count_ok@1
- SG_ok@1
- strict_valid@1
- 如果实现了真实 K=5，再报告对应 @5
- n_sites>=6 / n_sites>=12 / num_elements>=4 / extraction_hard breakdown

目的：

判断 geometry/free_params/lattice head 是否比当前 prototype/fallback 更有潜力。

Gate:

- 如果 GT-WA geometry 明显差，先不要跑 full generation；优先分析 free_param/lattice loss、target scale、wrapped coordinate、lattice normalization。

输出：

- `reports/symcif_v4_mp20_minicfjoint/04_gtwa_geometry_eval.md`
- `reports/symcif_v4_mp20_minicfjoint/04_gtwa_geometry_eval.json`

========================================
任务 5：constrained full generation，K<=5
========================================

只有任务 0、2.5、3、4 的 gate 通过后，才在 MP-20 val 上做完整生成。

input:

- formula + GT_SG

decode:

- constrained autoregressive row generation
- action 必须是当前 SG 下 legal `element@OrbitToken`
- element 只能来自 formula
- remaining composition 不能为负
- remaining composition 必须仍可 exact-cover
- stop 只能在 formula 完全闭合时允许
- canonical row order
- active free_params only
- lattice generated by lattice_head

K:

- only K=1 and K=5
- 不跑 match@20

必须输出：

- match@1 / match@5
- RMSE@1 / RMSE@5
- WA_hit@1 / WA_hit@5
- skeleton_hit@1 / skeleton_hit@5
- readable@1/@5
- formula_ok@1/@5
- atom_count_ok@1/@5
- SG_ok@1/@5
- strict_valid@1/@5
- failed_with_WA_hit / failed_without_WA_hit
- n_sites>=6 / n_sites>=12 / num_elements>=4 / extraction_hard breakdown

Baseline:

同时读取或复现当前 MP-20 val 的 SymCIF-v4 pipeline `WA5xgeom4` K<=5 指标，作为对照。

注意：

- baseline 必须是 MP-20 val，不允许拿 MP-20 test summary 直接对比。
- 不要用 test 调参。
- 如果 val baseline artifact 不存在，可以只复现这个 baseline；不要顺手扩大到 test、MPTS-52 或 CrystaLLM。

输出：

- `reports/symcif_v4_mp20_minicfjoint/05_full_generation_k5_eval.md`
- `reports/symcif_v4_mp20_minicfjoint/05_full_generation_k5_eval.json`

========================================
任务 6：最终决策报告
========================================

生成：

- `Log_GPT/round_YYYYMMDD_NN_mp20_minicfjoint/README.md`
- `Log_GPT/round_YYYYMMDD_NN_mp20_minicfjoint/comprehensive_mp20_minicfjoint_report.md`

最终报告必须回答：

1. v3 文本 CrystalFormer-like 的失败点是否被本实验避开？
2. GT roundtrip 是否足够干净？
3. small-overfit 是否通过？如果失败，失败在哪里？
4. Mini-CFJoint teacher-forcing 是否学到 W/A/X/L？
5. GT-WA geometry eval 是否优于或接近当前 geometry fallback？
6. full generation K<=5 是否优于当前 SymCIF-v4 val baseline？
7. n_sites>=6 / n_sites>=12 / extraction_hard 是否改善？
8. 如果失败，失败在：
   - action / orbit generation
   - element assignment
   - formula closure
   - free_params/lattice
   - constrained decoding
   - data规模/复杂子集
9. 下一步应该：
   - 继续 Mini-CFJoint
   - 改 geometry loss
   - 改 WA/action head
   - 增加 complex weighting
   - 加连续变量采样机制
   - 还是回到当前 pipeline

成功判定：

- 如果 Mini-CFJoint val match@1 或 match@5 比 current pipeline val baseline 高 >= 3 pp，且 RMSE 不恶化，继续该路线。
- 如果 GT-WA geometry 明显更好但 full generation 差，下一步修 W/A generation 或 constrained decoding。
- 如果 teacher-forcing 好但 generation 差，下一步修 exposure bias、beam search、legal action mask。
- 如果 GT-WA geometry 也差，下一步修 free_param/lattice loss、target normalization 和连续变量采样。
- 如果 small-overfit 失败，优先查实现 bug，不要扩大实验。

停止规则：

- 不跑 match@20。
- 不跑 MPTS-52。
- 不跑 CrystaLLM baseline。
- 不跑 MP-20 test。
- 不启动大模型。
- 不同时开多个修正分支。
- 不在 gate 失败后继续消耗大量 GPU/CPU 跑 full eval。
