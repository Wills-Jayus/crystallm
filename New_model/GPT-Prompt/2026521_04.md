你现在接手 SymCIF / CrystaLLM 项目。不要继续修 v2/v3 文本 sampler 的小问题；本轮要实现一个新的结构化前端：

Composition-Exact Wyckoff Table Generator / SymCIF-v4 frontend

目标：
给定完整 conventional cell formula + fixed space group，生成 by-construction formula-exact、SG-legal 的 Wyckoff-assignment table，然后补坐标和晶胞，最终渲染成标准 CIF。

非常重要的硬约束：
1. 用户输入 formula 必须解释为完整 conventional cell composition，例如 Ca4 F8，而不是 reduced formula CaF2。
2. 不需要用户输入 Z，不允许把 Z 当成模型输入，也不允许枚举 latent Z。
3. 可以从 full formula counts 内部派生 reduced_formula 和 z = gcd(counts)，但只能用于日志、分桶和统计，不能替代 target_counts。
4. exact-cover 的目标计数永远是 conventional cell formula_counts。
5. SG 永远是输入条件，不生成 SG。
6. 最终输出必须是 CIF。WA-table 只是中间产物，不能把任务完全变成 CrystalFormer。
7. 不要做事后 repair，不要扩大 pure rejection，不要继续训练 assignment_template_id 分类器。
8. 本轮重点是结构化输出空间和全局闭合，不是微调文本 token 顺序。

背景：
- 已有 `data/structured_symcif_v3`，每条记录包含 formula、formula_counts、sg、sg_symbol、z、skeleton_template_key、skeleton_template_uid、assignment、free_coords、lattice。
- 旧 skeleton ranker 的 `mlp_sg+formula` top5 = 80.6%，已经达到 train-catalog retrieval upper bound；所以继续只从 train template catalog 检索没有意义。
- 组合式 assignment generator 在 oracle skeleton 下 test assignment_top5 = 92.6%，assignment_top20 = 96.6%，closure = 100%；assignment 不是当前主瓶颈。
- 当前主要瓶颈是 unseen skeleton template，以及 WA-table 到 CIF 渲染后的 formula/SG/readability 一致性。
- v3 文本方案曾出现中间 formula closure = 100%，但最终 CIF formula_ok 只有 81.88%，所以本轮必须检查 rendered CIF-level formula_ok / SG_ok，而不是只检查中间 WA-table。

本轮总目标：
实现从

    full conventional formula + SG
    -> exact-cover skeleton / WA-table candidates
    -> WA-table scorer / ranker
    -> coords/lattice fallback
    -> rendered CIF
    -> closure diagnostics

的完整前端 pipeline。

不要一开始跑 full StructureMatcher match@5。先把离散前端和 CIF-level closure 做扎实。

C. Step 2: Exact-cover skeleton candidate generation

实现 src/symcif_v4/exact_cover.py

输入：

target_counts = full conventional formula_counts
sg
Wyckoff table for this SG

输出：

skeleton candidates
每个 skeleton 是 canonical sorted sequence of Wyckoff site tokens
candidate 不限制在 train catalog 里
允许生成训练集中没见过的 skeleton

合法 skeleton 条件：

所有 site 都来自输入 SG 的 Wyckoff table。
skeleton multiplicity sum 等于 sum(target_counts.values())。
skeleton multiplicities 必须可被 target_counts 分割，即至少存在一种 assignment，使每个元素的 assigned multiplicity sum 等于该元素 count。
fully fixed Wyckoff site 默认最多使用 1 次。
free-parameter Wyckoff site 可以重复使用，但要受总 atom count 限制。

实现要求：

使用 DFS + memoization / dynamic programming。
每一步按 canonical site order 生成，避免同一 multiset 的排列重复。
加 feasibility check：
对剩余 total count 和每个 element remaining count，检查是否还能被剩余 multiplicities exact-cover。
输出 candidate_count，并记录 timeout / explosion cases。
对复杂样本可设置 --max-candidates，但必须先统计 uncapped 或 high-cap coverage；如果 cap 截断了候选，必须在报告中记录 truncated=true。

推荐函数接口：

@dataclass(frozen=True)
class WyckoffSiteToken:
    sg: int
    letter: str
    multiplicity: int
    site_symmetry: str
    enumeration: int | str | None
    free_mask: tuple[bool, bool, bool]
    canonical_key: str
    is_fully_fixed: bool

@dataclass
class SkeletonCandidate:
    sg: int
    sites: list[WyckoffSiteToken]
    multiplicities: list[int]
    skeleton_key: str
    total_atoms: int

def enumerate_skeleton_candidates(
    target_counts: dict[str, int],
    sg: int,
    wyckoff_sites: list[WyckoffSiteToken],
    max_candidates: int | None = None,
    timeout_s: float | None = None,
) -> list[SkeletonCandidate]:

D. Step 3: WA-table candidate generation

实现 WA-table candidate generation。可以先 skeleton 后 assignment，也可以直接 action = element @ site。推荐先 skeleton 后组合式 assignment，因为已有 assignment generator 结果很好。

WA-table 定义：
一个 WA-table 是：

[
  (element, wyckoff_site_token),
  ...
]

合法 WA-table 条件：

每个 site 来自输入 SG。
对每个元素 e：
sum(multiplicity(site) for rows where element == e) == target_counts[e]
所有 site rows 的总 multiplicity 等于 conventional cell total atoms。
canonical order 稳定，方便和 GT 比较。

实现两种模式：

exhaustive assignment mode
对 skeleton candidate 枚举所有 formula-exact assignment。
用于 coverage 评估。
对候选爆炸样本允许 cap，但必须记录。
beam assignment mode
复用或改造已有 scripts/train_combinatorial_assignment_generator.py
给定 skeleton，用组合式 assignment generator 输出 topK assignments。
用于 ranked candidate pipeline。

推荐函数接口：

@dataclass
class WATableCandidate:
    sg: int
    formula_counts: dict[str, int]
    rows: list[tuple[str, WyckoffSiteToken]]
    skeleton_key: str
    assignment_key: str
    wa_key: str
    source: str
    score: float | None = None

def enumerate_wa_tables_for_skeleton(
    target_counts: dict[str, int],
    skeleton: SkeletonCandidate,
    max_assignments: int | None = None,
) -> list[WATableCandidate]:
    ...

GT comparison：
从 structured record 的 skeleton_template_key 和 assignment 构造 GT WA key。注意 canonical site order 必须和已有结构化数据定义保持一致：

(multiplicity, letter, enumeration, site_symmetry, original_index)

如果 original_index 不存在，则使用稳定 canonical order，并在报告中说明。

F. Step 5: WA-table scorer / ranker

实现 scripts/train_wa_table_scorer.py

目标：
从 exact-cover candidates 中学习排序：

score(WA-table | full conventional formula_counts, SG)

注意：

不能只输出 train skeleton_template_id。
可以使用训练集中统计的频率作为 feature，但候选空间必须来自 exact-cover generator。
必须能给 unseen skeleton 打分。

实现两个 scorer：

Baseline scorer
使用可解释频率 / 统计特征：
SG prior
site token frequency
multiplicity pattern frequency
element@site frequency
formula anonymous pattern frequency
nsites prior
penalty for too many repeated free sites
输出 topK 排序。
Neural scorer
简单即可，不要过度复杂：
formula count vector embedding
SG embedding
WA rows encoded as set / multiset
row feature:
element embedding
multiplicity embedding
letter embedding
site_symmetry embedding
enumeration embedding
free_mask embedding
aggregation:
DeepSets pooling，或 Transformer encoder without positional encoding
output scalar score
train with candidate set CE / sampled negatives / pairwise ranking loss

训练数据：

positive = GT WA-table
negatives = same sample 的其他 exact-cover WA candidates
如果 GT 不在 candidates，跳过训练并记录。
每个样本 negative 数可以 cap，例如 64 / 128。
为了避免候选爆炸，训练时使用 sampled negatives；评估时使用 full or high-cap candidate list。

输出：

runs/wa_table_scorer_v1/ckpt.pt
reports/composition_exact_v1/wa_scorer_eval_summary.json
reports/composition_exact_v1/test_wa_predictions.jsonl

评估指标：

skeleton_top1/top5/top20 from ranked WA candidates
wa_top1/top5/top20
gt_skeleton_in_ranked_topK
gt_wa_in_ranked_topK
candidate_nonempty_rate
rendered-ready topK count
per SG / per nsites breakdown

验收标准：

WA top20 必须超过当前 joint_pair_top20 = 76.8%。
skeleton top20 应显著突破 80.6% train-catalog ceiling；如果 exact-cover coverage 已经高但 scorer top20 低，说明 ranker 不够。
输出 baseline scorer 和 neural scorer 对比，不要只报一个模型。

G. Step 6: CIF rendering

实现 scripts/render_composition_exact_cifs.py

输入：

ranked WA candidates
structured dataset records
mode:
gt_oracle
retrieval
default_safe

输出：

CIF files or JSONL with CIF string
render metrics

必须支持：

coord_mode=gt_oracle
用途：诊断 render 链路，不作为最终生成方法。
如果 WA-table 等于 GT WA-table，则直接使用 GT free_coords 和 GT lattice。
如果 WA-table 不等于 GT，则尽量按 same site token 映射 GT coords，缺失部分用 default_safe。
报告中必须标记 oracle。
coord_mode=retrieval
用途：非 oracle fallback。
检索优先级：
same SG + same WA key from train
same SG + same skeleton_key from train
same SG + same formula anonymous pattern
same crystal system / SG median lattice
坐标：
对相同 site token，使用 retrieved free coords。
fixed coords 由 Wyckoff table/template 提供。
没检索到的 free coords 用 deterministic default，例如 0.125 / 0.25 / 0.375，并加小的 candidate-index deterministic offset，避免重复重合。
保证 fractional coords wrap 到 [0,1)。
coord_mode=default_safe
用途：smoke test。
lattice 使用每个 SG 或 crystal system 的 train median。
coords 用 Wyckoff fixed template + deterministic free defaults。
目标是 CIF readable / formula_ok / SG_ok，不追求 match。

渲染要求：

优先复用 src/symcif_v3/to_cif.py。
使用 Structure.from_spacegroup 或现有可靠实现展开。
CIF 必须包含：
_chemical_formula_sum
_symmetry_Int_Tables_number
_symmetry_space_group_name_H-M
cell lengths / angles
atom site loop
渲染后重新用 pymatgen 读取，并计算：
readable
formula_ok against full conventional formula_counts
SG_ok against input SG
composition exactness
number of atoms after expansion
to_cif_success
duplicate/overlap warning if available

输出：

reports/composition_exact_v1/render_cif_summary.json
reports/composition_exact_v1/render_cif_examples/
reports/composition_exact_v1/rendered_test_topk.jsonl

验收标准：

在 default_safe 或 retrieval 模式下，topK rendered CIF 的 readable 应接近或超过 v3 文本版。
formula_ok 必须接近 100%；如果不到 98%，说明 WA-table -> CIF render 有系统问题，需要定位。
SG_ok 尽量接近 100%；如果 spglib 识别不稳定，必须区分：
WA-table SG legal
rendered CIF spglib SG_ok
evaluator SG_ok
不要把中间 closure 当作最终 closure；必须报告 CIF-level closure。
